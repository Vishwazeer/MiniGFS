Atomic Record Append

1

Client requests append to Master

The client contacts the Master node to find which chunkservers hold the target file's last chunk.

CLIENT → MASTER: APPEND(data, filename)
MASTER → CLIENT: Primary + Replica locations
2

Client sends data to ALL replicas

Data is pushed through a pipeline chain, not broadcast. Each node forwards to the next, minimizing client bandwidth.

Pipeline: Client → CS-03 → CS-07 → CS-12

3

Primary picks offset, issues WRITE

The primary chunkserver (CS-03) selects a byte offset and instructs all replicas to write at the identical position.

CS-03 (Primary): OFFSET SELECTED: byte 1,048,576
CS-03 → CS-07, CS-12: WRITE at offset 1,048,576
4

All ACK → Primary ACKs Client

Each replica acknowledges. Once all ACKs arrive at the primary, the primary confirms success to the client.

CS-07: ACK ✓   CS-12: ACK ✓ → CS-03: ALL ACK ✓ → CLIENT: SUCCESS
ATOMIC: Either all replicas wrote, or none did
CS-07 FAILS MID-WRITE
CS-03 receives only partial ACKs
ABORT signal sent to all replicas
WRITE ROLLED BACK — Client retries from Step 1
Consistency guaranteed ✓

Tech Stack

C++

C++ Core Engine

High-performance systems programming for zero-overhead abstractions. Master node, chunkserver, and client library all implemented in modern C++17 with STL concurrency primitives.

gR

gRPC Communication

Bidirectional streaming RPC for master-client coordination and client-chunkserver data transfer. Enables pipelined data transfer achieving 2.1 GB/s aggregate throughput.

PB

Protocol Buffers

Efficient binary serialization for all inter-node messages. ChunkHandle, MutationOrder, and HeartbeatResponse all defined in .proto schemas — compact, versioned, language-agnostic.

⬡

Consistent Hashing

Custom ring implementation with 3 virtual nodes per server for uniform load distribution. Minimizes chunk remapping on node addition/removal to under 6% disruption.
